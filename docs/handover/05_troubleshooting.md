# 05. トラブルシューティング

> 最終更新: 2026-06 ／ 実際に起きた事故・判明済みの罠を「症状 → 原因 → 対処」で。番号は他資料から参照される。

## 1. GASのコードを変えたのにAPIの応答が変わらない

- **原因（頻出）**: Webアプリの再デプロイで「バージョン: **新バージョン**」を選んでいない（既存バージョンのまま「デプロイ」を押しても何も変わらない）。または「**新しいデプロイ**」を作ってしまい別URLが発行され、既存URLは旧版のまま
- **対処**: デプロイ → **デプロイを管理** → 既存ウェブアプリの鉛筆 → バージョンのプルダウンで「**新バージョン**」を選択 → デプロイ（2026-06-12 に実際に空振りした事例あり）
- 補足: フロント `/api/data` に60秒キャッシュがあるため、正しく再デプロイしても最大1分は旧データが返る

## 2. 売上が¥0で表示される

- **原因**: `M_月次ボーナス` のD列（iriam実額）が未入力のまま「① CSV取込&ダッシュボード更新」を実行した（D列未入力=0で上書きする仕様）
- **対処**: D列を入力 → 即時反映（SUMIFS参照）。反映しない場合は「②-2 全月一括 売上再同期」

## 3. clasp push したら本番の関数・ファイルが消えた

- **原因**: `clasp push --force` はローカルに無いファイルを本番から**削除**する
- **対処（復旧）**: このリポジトリの `gas/` から復元して push。またはGASエディタの「バージョン履歴」から戻す
- **予防**: 必ず `clasp pull` → 該当ファイルだけ編集 → `clasp push` の順。`--force` は使わない

## 4. バナイベの数値が日中どんどん増える

- **原因ではない**: 進行中の回（火曜開始〜月曜終了）はライブで応援ptが増える。**仕様**
- 確定値が必要なら終了済みの回・月で見る

## 5. ダッシュボードの入賞数とスプシ「DB_バナー実績集計」が合わない

- **原因**: 判定ロジックの違い。ダッシュボードAPI = **rank≤100で判定（正）**／スプシ集計（banner_summary.js）= 元データR列の入賞フラグ由来で、**最新週のフラグが不完全**なことがある
- **対処**: ダッシュボード側の数字を正とする

## 6. ライバー別でptが大きいのに順位が悪い（pt↔順位の逆転）

- **原因**: banner_active のO列（BlockId）が空で、同一EventId内の複数ブロック（=別ランキング）を分離できない。多人数事務所で顕著
- **対処**: 現状は仕様（画面にも注記あり）。OMNIA側でBlockIdが入ればGAS側コード（events構造）は対応済みで、即ブロック分離可能

## 7. 「🌙 ②-3 全月自動同期」がタイムアウトする

- **原因**: 既知（トリガー連鎖の各ステップが30分制限に当たる。根本原因未特定）
- **対処（回避手順）**: GASエディタから月別関数を順に手動実行する
  1. `syncToPL2026_01` → `syncToPL2026_02` → …（対象月分。各1〜2分）
  2. `applyPlBackgrounds`
  3. `rewriteLabelRevenueFormulas`

## 8. RAW_ライバー月次に同じ月のデータが重複している

- **経緯**: 過去に upsert の対象月比較バグ（Date型）で重複取込が発生（**修正済み**）
- **対処（再発時）**: GASエディタで `checkRawSummary`（月×事務所の件数一覧）→ `deleteRawRowsByMonthOffice('YYYY-MM', '事務所名')` で該当分を削除 → CSV再取込

## 9. ローカル `npm run dev` で「GAS_API_URL not set」

- **原因**: `.env.local` 未作成（gitignoreされているためcloneに含まれない）
- **対処**: [01_frontend.md](01_frontend.md) のローカル開発手順どおり `.env.local` を作成

## 10. Vercelのプレビューデプロイが開けない

- **原因**: プレビューURLは Vercel SSO 保護（仕様）
- **対処**: 動作確認はローカル dev か、main マージ後の本番で行う
